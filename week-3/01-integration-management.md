# Integration Management — The PM's Core Responsibility

## Why Integration Management Matters

Integration Management is the **only knowledge area that belongs exclusively to the project manager**. While subject matter experts handle scope, schedule, cost, and quality within their domains, only the PM integrates all these pieces into a coherent whole. On the exam, Integration questions test whether you understand the PM's unique role as the integrator — the person who sees the big picture and makes trade-off decisions.

> 🎯 **Exam Tip:** When a question asks "What should the PM do FIRST?" and the scenario involves multiple competing constraints, the answer often involves an Integration Management process (especially Perform Integrated Change Control).

---

## The 7 Integration Management Processes

| Process | Process Group | Key Output |
|---------|--------------|------------|
| Develop Project Charter | Initiating | Project Charter |
| Develop Project Management Plan | Planning | Project Management Plan |
| Direct and Manage Project Work | Executing | Deliverables, Work Performance Data |
| Manage Project Knowledge | Executing | Lessons Learned Register |
| Monitor and Control Project Work | Monitoring & Controlling | Work Performance Reports, Change Requests |
| Perform Integrated Change Control | Monitoring & Controlling | Approved/Rejected Change Requests |
| Close Project or Phase | Closing | Final Product, Transition, Lessons Learned |

## 1. Develop Project Charter

### What It Is

The project charter **formally authorizes** the project and gives the PM authority to apply organizational resources. Without a charter, the project doesn't officially exist.

### Key Inputs

- **Business case** — why are we doing this? (ROI, strategic alignment)
- **Benefits management plan** — how will we measure success?
- **Agreements** — contracts, MOUs, SOWs (for external projects)
- **Enterprise environmental factors (EEFs)** — market conditions, regulations
- **Organizational process assets (OPAs)** — templates, historical data

### What the Charter Contains

- Project purpose and justification
- Measurable objectives and success criteria
- High-level requirements and boundaries
- Overall project risk
- Summary milestone schedule
- Pre-approved financial resources (budget summary)
- Key stakeholder list
- PM assignment and authority level
- Name and authority of the sponsor

> 🎯 **Exam Tip:** The **sponsor** creates/issues the charter, NOT the PM. The PM may help write it, but the sponsor signs it. If a question asks who authorizes the charter — it's always the sponsor (or someone with equivalent authority).

### Exam Scenario: Charter Questions

**Scenario:** A PM is assigned to a new project but hasn't received a charter yet. The sponsor asks them to start planning immediately. What should the PM do?

**Answer:** Request that the charter be issued first. Without a charter, the PM has no formal authority. Starting work without authorization violates PMI's process framework.

**Scenario:** Midway through a project, a senior VP asks the PM to add a major feature. The VP outranks the sponsor. What should the PM do?

**Answer:** Direct the VP to the sponsor or submit a change request through Integrated Change Control. Authority comes from the charter, not org hierarchy. The PM works for the project, not for whoever has the highest title.

## 2. Develop Project Management Plan

### What It Is

The project management plan is the **master document** that defines how the project will be executed, monitored, controlled, and closed. It's not just a schedule — it's the collection of all subsidiary plans.

### Subsidiary Plans (Know These)

| Plan | Covers |
|------|--------|
| Scope Management Plan | How scope will be defined, validated, controlled |
| Requirements Management Plan | How requirements will be collected, documented, managed |
| Schedule Management Plan | How schedule will be developed, managed, controlled |
| Cost Management Plan | How costs will be estimated, budgeted, controlled |
| Quality Management Plan | Quality standards, metrics, and improvement processes |
| Resource Management Plan | How resources will be acquired, developed, managed |
| Communications Management Plan | Who gets what info, when, how |
| Risk Management Plan | How risks will be identified, analyzed, responded to |
| Procurement Management Plan | How procurement will be conducted |
| Stakeholder Engagement Plan | Strategies for engaging stakeholders |

### Baselines (The "Triple Constraint" Baselines)

- **Scope baseline** = Scope statement + WBS + WBS dictionary
- **Schedule baseline** = Approved schedule (with start/finish dates)
- **Cost baseline** = Time-phased budget (S-curve)

> 🎯 **Exam Tip:** Baselines can ONLY be changed through formal change control (Perform Integrated Change Control). If someone asks "can we adjust the baseline?" — the answer always involves a change request and CCB approval.

### Exam Scenario: PM Plan Questions

**Scenario:** A team member asks the PM how conflicts between schedule and cost will be resolved. Where should they look?

**Answer:** The Project Management Plan — specifically, it should define the priority/trade-off approach among constraints. Many PMs document this as a constraint priority matrix (e.g., Schedule is fixed, Cost is flexible, Scope is negotiable).

## 3. Direct and Manage Project Work

### What It Is

This is where the **actual work gets done**. The PM leads the team in executing the work defined in the PM plan, producing deliverables, and collecting work performance data.

### Key Activities

- Execute activities in the PM plan
- Create project deliverables
- Collect work performance data (raw measurements)
- Implement approved changes, corrective actions, preventive actions, and defect repairs
- Manage resources and stakeholder engagement
- Apply lessons learned from earlier phases

### Key Outputs

- **Deliverables** — tangible outputs (code, documents, hardware)
- **Work performance data** — raw, unanalyzed measurements (actual costs, actual dates, % complete)
- **Change requests** — generated when variances are found
- **Issue log updates**
- **Lessons learned register updates**

> 🎯 **Exam Tip:** Understand the data → information → reports flow:
>
> - **Work Performance Data** (raw) → produced in Direct & Manage
> - **Work Performance Information** (analyzed) → produced in individual control processes
> - **Work Performance Reports** (formatted) → produced in Monitor & Control Project Work

---

## 4. Manage Project Knowledge

### What It Is

Uses existing knowledge and creates new knowledge to achieve project objectives. This is about **lessons learned** — both using past lessons and capturing new ones.

### Two Types of Knowledge

| Type | Description | Example |
|------|-------------|---------|
| **Explicit knowledge** | Documented, codified, easily shared | Procedures, templates, reports |
| **Tacit knowledge** | Personal experience, hard to articulate | Intuition, expertise, "tribal knowledge" |

### Key Concept: Lessons Learned

- Lessons learned are captured **throughout the project**, not just at the end
- The lessons learned register is a living document updated continuously
- At project close, lessons are transferred to the organizational knowledge base (OPA)

> 🎯 **Exam Tip:** If a question mentions "a similar project had this problem before" — the answer likely involves consulting lessons learned or organizational process assets.

---

## 5. Monitor and Control Project Work

### What It Is

Tracking, reviewing, and reporting overall project progress against the PM plan. This is the "big picture" monitoring process — it looks across all knowledge areas.

### Key Activities

- Compare actual performance against the PM plan
- Assess whether corrective/preventive actions are needed
- Monitor implementation of approved changes
- Provide forecasting information (EAC, ETC)
- Report project status to stakeholders

### Key Outputs

- **Work performance reports** — formatted status reports, dashboards, memos
- **Change requests** — when variances require plan changes

### Exam Scenario: Monitor & Control

**Scenario:** The PM notices that the project is 15% over budget at the halfway point. What should they do?

**Answer:** Analyze the variance (is it a one-time issue or a trend?), determine root cause, generate a change request if the cost baseline needs updating, and report the variance with forecasts (EAC) to stakeholders. Don't just absorb the overrun silently.

## 6. Perform Integrated Change Control (ICC)

### What It Is

The process of reviewing ALL change requests, approving/rejecting changes, and managing changes to deliverables, project documents, and the PM plan. This is the **gatekeeper process** for the entire project.

> 🎯 **Exam Tip:** This is one of the MOST tested processes on the PMP exam. Know the CCB workflow cold.

### The Change Control Board (CCB)

The CCB is a formally constituted group responsible for reviewing, evaluating, approving, deferring, or rejecting changes. Key facts:

- The PM may or may not be a member (depends on the organization)
- The PM always has the right to approve minor changes within their authority level
- The sponsor or CCB approves changes that affect baselines
- The CCB composition is defined in the Change Management Plan

### CCB Process — Step by Step

```text
1. Change is identified (by anyone — PM, team, stakeholder, customer)
         ↓
2. Change request is documented (formally, in writing)
         ↓
3. Impact assessment is performed
   - Schedule impact?
   - Cost impact?
   - Scope impact?
   - Quality impact?
   - Risk impact?
         ↓
4. Change request is submitted to CCB for review
         ↓
5. CCB evaluates and decides:
   - APPROVE → implement the change, update baselines/plans
   - REJECT → document reason, notify requestor
   - DEFER → revisit later with more information
         ↓
6. If approved:
   - Update project management plan and baselines
   - Update project documents
   - Communicate to affected stakeholders
   - Implement through Direct & Manage Project Work
         ↓
7. Verify implementation (through quality control)
         ↓
8. Document in change log
```

### Types of Changes

| Type | Description | Example |
|------|-------------|---------|
| **Corrective action** | Realigns future performance with the plan | Adding resources to recover schedule |
| **Preventive action** | Ensures future performance stays aligned | Adding a review gate to prevent defects |
| **Defect repair** | Fixes a non-conforming deliverable | Fixing a bug, correcting a document |
| **Updates** | Changes to formally controlled documents | Updating the risk register |

### Configuration Management

Part of ICC — ensures the integrity of project artifacts:

- **Configuration identification** — selecting and identifying configuration items
- **Configuration status accounting** — recording and reporting item status
- **Configuration verification and audit** — ensuring items are complete and correct

> 🎯 **Exam Tip:** NEVER implement a change without going through ICC — even if the change seems small, even if the customer is demanding it, even if your boss tells you to "just do it." The PMI-correct answer is ALWAYS to follow the change control process.

### Exam Scenarios: ICC

**Scenario:** A customer calls the PM directly and says "I need this small feature added by Friday." What should the PM do?

**Answer:** Thank the customer, document the request as a formal change request, perform impact assessment, and submit to the CCB (or approve if within PM's authority level). Never say "sure, we'll add it" without following the process.

**Scenario:** The PM discovers that a team member implemented a change requested by a stakeholder without going through change control. What should the PM do FIRST?

**Answer:** Evaluate the impact of the unauthorized change. Then determine whether to submit a change request retroactively or reverse the change. Also address the process violation with the team member (coaching, not punishment — PMI mindset).

**Scenario:** A change request is approved but the PM realizes during implementation that it will cost 3x more than estimated. What should the PM do?

**Answer:** Stop implementation, reassess the impact, and submit a new change request with updated estimates. An approved change can be revisited if the assumptions behind the approval have changed.

## 7. Close Project or Phase

### What It Is

Finalizing all activities across all process groups to formally complete the project or phase. This is the PM's last responsibility.

### Key Activities

- Confirm all work is complete and deliverables are accepted
- Transfer the final product/service/result to the customer or operations
- Collect and archive all project documents
- Finalize lessons learned and transfer to organizational knowledge base
- Release project resources
- Close out all procurement contracts
- Perform final financial closure (final payments, close accounts)
- Obtain formal sign-off from the sponsor/customer
- Celebrate success with the team

### What Gets Archived

- Project management plan (final version)
- All project documents
- Lessons learned register
- Change log
- Risk register (for future reference)
- Final deliverables documentation

> 🎯 **Exam Tip:** Projects can be closed for multiple reasons:
>
> - Successfully completed
> - Cancelled (sponsor decision, business case no longer valid)
> - Terminated early (legal issues, force majeure)
>
> Regardless of WHY a project closes, you STILL perform the Close Project process. Even cancelled projects need formal closure.

### Exam Scenario: Close Project

**Scenario:** A project is cancelled by the sponsor due to budget cuts. The PM is told to "just stop work." What should the PM do?

**Answer:** Perform formal project closure — document what was completed, archive records, capture lessons learned, release resources, close contracts, and get formal acknowledgment of cancellation. Never just walk away from a project.

**Scenario:** The project is complete and the customer has accepted all deliverables. What is the PM's LAST action?

**Answer:** Transfer lessons learned to the organizational process assets (knowledge base). This ensures future projects benefit from this project's experience. Note: Some questions frame this as "release resources" being last — but PMI emphasizes knowledge transfer as the final intellectual activity.

## Integration Management — Summary Decision Framework

When you see an exam question about Integration, ask yourself:

1. **Is this about starting the project?** → Charter
2. **Is this about planning how to do the work?** → PM Plan
3. **Is this about doing the work?** → Direct & Manage
4. **Is this about using/creating knowledge?** → Manage Knowledge
5. **Is this about tracking progress?** → Monitor & Control
6. **Is this about a change?** → Perform ICC
7. **Is this about finishing?** → Close Project

## Common Exam Traps — Integration Management

| Trap | Correct Answer |
|------|---------------|
| "The sponsor wants to skip the charter for a small project" | Still need a charter — it authorizes the PM |
| "A stakeholder says the change is too small for paperwork" | All changes go through ICC, regardless of size |
| "The PM should update the baseline to reflect actual performance" | Baselines only change through approved change requests |
| "Lessons learned are captured at project end" | They're captured throughout — continuously |
| "The PM can approve all changes" | Only within their defined authority level |
| "The project is cancelled so no closure is needed" | Always perform formal closure, even for cancelled projects |
